Aula 01 · Git Essencial
Exercícios Práticos

Pratique os comandos aprendidos em sala. Complete todos os exercícios antes da Aula 02. Dica: sempre use git status entre os comandos para observar o que está acontecendo.

Fácil
EX-01
Configurando sua identidade

Configure o Git com suas informações reais e verifique o resultado.

1Configure seu nome completo com git config --global user.name
2Configure seu email institucional com git config --global user.email
3Configure o VS Code como editor padrão com core.editor "code --wait"
4Execute git config --list e confirme que as três configurações aparecem.
✓ Resultado esperado

git config --list deve exibir user.name, user.email e core.editor com seus valores.

EX-02
Primeiro repositório e commit

Crie um repositório do zero e faça seu primeiro commit semântico.

1Crie uma pasta projeto-ods e entre nela
2Execute git init e verifique a pasta .git/ criada
3Crie um arquivo README.md com o nome do projeto e uma descrição
4Use git add README.md e depois git status
5Faça o commit: git commit -m "docs: adiciona README inicial"
6Confira com git log --oneline
✓ Resultado esperado

git log --oneline mostra 1 commit com a mensagem semântica correta.

EX-03
Commits semânticos variados

Pratique usar tipos diferentes de commits semânticos em um projeto real.

1No projeto-ods, crie os arquivos index.html e styles.css
2Commit do HTML: "feat: cria estrutura HTML da página inicial"
3Commit do CSS: "style: adiciona estilos base da página"
4Adicione um comentário ao HTML e commit: "docs: documenta seções do HTML"
5Corrija um erro de digitação qualquer: "fix: corrige typo no título"
6Visualize com git log --oneline
✓ Resultado esperado

Histórico com ao menos 5 commits, todos com tipos semânticos diferentes e descrições claras.

Médio
EX-04
Add seletivo e Staging Area estratégica

Pratique adicionar arquivos seletivamente, criando commits coesos e bem separados.

1Modifique simultaneamente: index.html, styles.css e crie script.js
2Adicione somente o script.js ao staging
3Verifique que o git status mostra staged e unstaged corretamente
4Commit: "feat: adiciona funcionalidade JavaScript"
5Agora adicione as mudanças de HTML e CSS juntas e faça 1 commit
6Explique (em comentário ou papel): por que separar esses commits é melhor?
✓ Resultado esperado

Dois commits separados, cada um com contexto claro. O git log conta uma história coerente.

EX-05
Desfazendo com git reset

Pratique os três modos do git reset e observe as diferenças.

1Crie 2 novos arquivos e faça 2 commits (um por arquivo)
2Use git reset --soft HEAD~1 — observe o git status
3Refaça o commit com uma mensagem melhor
4Crie mais 1 commit qualquer
5Use git reset --mixed HEAD~1 — observe a diferença no git status
6Documente (comentário): qual a diferença prática entre --soft e --mixed?
$ git log --oneline # antes
$ git reset --soft HEAD~1
$ git status # compare!
✓ Resultado esperado

Entender claramente quando usar --soft (refazer mensagem) vs --mixed (reestruturar commits).

Difícil
EX-06
Squash manual de commits com git reset

Você fez 4 commits pequenos que deveriam ser 1 commit único e bem descrito. Use git reset para consolidá-los sem perder o código.

1Crie uma sequência de 4 commits temáticos mas fragmentados, ex: "wip", "corrige", "esqueci disso", "pronto agora"
2Use git log --oneline para confirmar os 4 commits
3Execute git reset --soft HEAD~4 para desfazê-los mantendo os arquivos staged
4Verifique com git status que todas as mudanças ainda estão na Staging Area
5Crie 1 único commit com uma mensagem semântica profissional que descreve todas as mudanças
6Confirme: git log --oneline mostra apenas 1 commit no lugar dos 4 anteriores
💡 Por que isso é importante?
Antes de abrir um PR profissional, é comum "limpar" commits de desenvolvimento em poucos commits coesos. Isso é chamado de squash — você fará isso com frequência no projeto ODS.
⚡ Desafio extra

Tente consolidar os 4 commits em 2 commits lógicos em vez de 1. Use git add seletivo após o reset.

Aula 02 · Branches
Exercícios Práticos

Pratique o fluxo de branches individualmente e, nos exercícios médio e difícil, simule colaboração com um colega de turma — combinando quem faz cada etapa.

Fácil
EX-01
Primeiro ciclo de feature branch

Execute o fluxo completo de uma branch de feature do início ao fim.

1No projeto-ods, crie e entre na branch feature/sobre-o-projeto
2Crie o arquivo sobre.html com algum conteúdo
3Faça 2 commits semânticos nessa branch
4Volte para o main e verifique que o sobre.html não existe nele
5Faça o merge: git merge feature/sobre-o-projeto
6Confirme que o arquivo agora existe no main. Delete a branch com -d
✓ Resultado esperado

O arquivo aparece no main após o merge. O git log --oneline mostra os commits que vieram da feature branch.

EX-02
Nomenclatura e navegação entre branches

Pratique criar branches com nomenclatura correta e navegar entre elas.

1A partir do main, crie 4 branches: feature/mapa-ods, fix/calculo-meta, docs/guia-contribuicao, hotfix/seguranca-csrf
2Liste todas as branches com git branch
3Navegue para cada uma e crie 1 arquivo diferente em cada
4Faça 1 commit em cada branch
5Tente criar uma branch com nome inválido: feature/Mapa ODS — o que acontece?
✓ Resultado esperado

4 branches com nomenclatura correta, cada uma com 1 arquivo e 1 commit. Branch com nome inválido gera erro.

EX-03
Visualizando o grafo de branches

Crie uma estrutura de branches e visualize o histórico em formato de grafo.

1Faça 2 commits no main
2Crie e trabalhe em feature/a (2 commits), depois volte ao main
3Faça 1 commit no main
4Crie e trabalhe em feature/b (1 commit), volte ao main
5Merge da feature/a, depois merge da feature/b
6Execute git log --oneline --graph --all --decorate
$ git log --oneline --graph --all --decorate
✓ Resultado esperado

O grafo mostra duas bifurcações e merges claramente, com os nomes das branches.

Médio
EX-04
Fast-forward vs Merge Commit

Provoque os dois tipos de merge e compare o histórico resultante.

1Fast-forward: Crie feature/ff a partir do main atual. Faça 2 commits nela. Não faça nada no main enquanto isso. Volte ao main e faça o merge.
2Observe o git log: sem merge commit, histórico linear
3Merge commit: Crie feature/mc a partir do main. Faça 1 commit nela. Volte ao main e faça mais 1 commit. Agora faça o merge.
4Observe o git log --graph: aparece o merge commit com dois pais
5Documente a diferença em um comentário no README
✓ Resultado esperado

Você consegue explicar quando cada tipo de merge ocorre e identificar a diferença no git log --graph.

EX-05
Provocar e resolver um conflito (dupla)

Em dupla, simulem a situação mais comum de conflito em projetos colaborativos.

1Ambos partem do mesmo estado do main
2Pessoa A: cria feature/titulo-a e altera a linha 1 do index.html
3Pessoa B: cria feature/titulo-b e altera a mesma linha 1 com texto diferente
4Merge da feature/titulo-a no main (sem conflito)
5Tente merge da feature/titulo-b — o conflito vai aparecer!
6Resolva o conflito juntos, mantendo o melhor dos dois textos
✓ Resultado esperado

Conflito resolvido com sucesso. O git log mostra o merge commit com dois pais corretamente.

Difícil
EX-06
Duas features paralelas com múltiplos conflitos (trio)

Simule um cenário de desenvolvimento real com três pessoas em paralelo, com múltiplos conflitos para resolver.

1Pessoa A (líder): fica no main e faz 2 commits adicionando estrutura base ao index.html
2Pessoa B: cria feature/header, modifica a div de header do HTML e faz 3 commits
3Pessoa C: cria feature/footer, modifica a div de footer E uma linha do header. Faz 2 commits
4Pessoa A faz merge da feature/header → ok ou resolve conflitos
5Pessoa A faz merge da feature/footer → vai ter conflito no header!
6Resolvam juntos: decidam qual versão do header fica, mantenham o footer
7Resultado final: git log --graph --oneline --all mostrando tudo
💡 Reflexão pós-exercício
Como a comunicação entre a equipe pode evitar conflitos antes mesmo de acontecerem? O que vocês combinariam de fazer diferente?
Aula 03 · GitHub
Exercícios Práticos

Para esses exercícios, você precisará de uma conta no GitHub. Faça os exercícios em pares — um abre o PR, o outro faz o code review.

Fácil
EX-01
Publicar repositório no GitHub

Conecte seu repositório local ao GitHub e publique todos os commits.

1Crie um repositório vazio no GitHub (sem README, sem .gitignore)
2Copie a URL HTTPS do repositório
3No terminal: git remote add origin URL
4Execute git remote -v para confirmar a conexão
5Execute git push -u origin main
6Acesse o repositório no GitHub e confirme que todos os commits estão visíveis
✓ Resultado esperado

Repositório visível no GitHub com todo o histórico de commits da Aula 01 e 02.

EX-02
Primeiro Pull Request

Abra seu primeiro PR no GitHub com título profissional.

1Crie localmente a branch feature/contato e adicione um arquivo contato.html
2Faça 2 commits semânticos
3Execute git push -u origin feature/contato
4No GitHub, clique em "Compare & pull request"
5Defina o título: feat(contato): adiciona página de contato
6Escreva uma descrição breve explicando o que o PR faz e por quê
7Adicione um colega como revisor e abra o PR
✓ Resultado esperado

PR aberto com título semântico correto, descrição e revisor adicionado.

EX-03
Respondendo a um code review

Pratique o ciclo completo de receber feedback e atualizar o PR.

1O colega revisor deve fazer pelo menos 2 comentários no PR do EX-02 pedindo melhorias
2Leia todos os comentários antes de fazer qualquer mudança
3Faça as mudanças solicitadas nos arquivos localmente
4Commit: "fix(contato): aplica feedback do code review"
5Execute git push — o commit aparece automaticamente no PR
6Responda cada comentário no GitHub explicando o que foi feito
✓ Resultado esperado

PR atualizado com novos commits. Todos os comentários respondidos. Revisor aprova.

Médio
EX-04
Clone, branch e PR completo

Simule entrar em um projeto existente como novo contribuidor.

1Clone o repositório do colega (não o seu!) com git clone URL
2Crie a branch feature/seu-nome/melhoria
3Faça uma melhoria real no projeto (adicione conteúdo, corrija algo)
43 commits semânticos
5Push e abra PR com título profissional
6Preencha a descrição com: o quê foi feito, por quê e como testar
✓ Resultado esperado

PR aberto no repositório do colega, com branch no padrão correto e descrição completa.

EX-05
fetch, pull e resolução de divergência

Pratique sincronizar com o remoto quando há commits divergentes.

1Em dupla: Pessoa A e Pessoa B clonam o mesmo repositório
2Pessoa A faz 1 commit e push no main
3Pessoa B (sem fazer pull) faz 1 commit diferente no main local
4Pessoa B tenta git push — vai ser rejeitado. Leia o erro.
5Pessoa B executa git fetch e depois git log HEAD..origin/main para ver o que está no remoto
6Pessoa B executa git pull, resolve eventual conflito e faz push
✓ Resultado esperado

Entender por que o push foi rejeitado e como resolver com pull antes de tentar novamente.

Difícil
EX-06
Fluxo profissional completo com review e CI (trio)

Simule um sprint real: três pessoas, repositório compartilhado, dois PRs simultâneos com review cruzado.

1Setup: Pessoa A cria o repositório no GitHub como organização/dono, adiciona B e C como colaboradores
2Paralelo: B trabalha em feature/b/nova-secao e C trabalha em feature/c/menu-navegacao, ambos em paralelo
3PRs: Ambos abrem PRs ao mesmo tempo, pedindo review cruzado (B revisa C, C revisa B)
4Reviews: Cada um faz pelo menos 2 comentários técnicos no PR do outro
5Iteração: Ambos respondem aos comentários e fazem os ajustes com novos commits
6Merge: A aprova e faz merge do PR de B. Isso pode gerar conflito no PR de C — resolvam!
7A aprova e faz merge do PR de C. Repositório final com ambas as features integradas.
💡 Bônus: GitHub Actions
Se quiser ir além, configure um workflow simples em .github/workflows/check.yml que roda echo "Build OK" em cada PR. Observe o ✓ aparecer nos PRs.
🏆 Desafio Final Integrador
Operação ODS: Novo Desenvolvedor

Você acabou de entrar como desenvolvedor voluntário no Projeto ODS da instituição. A equipe do 2º Ano já estruturou o repositório e o fluxo de trabalho. Sua missão é entregar uma funcionalidade real seguindo todos os processos profissionais aprendidos nas três aulas — do git init ao merge aprovado.

⏱ ~90 minutos 👥 Individual ou dupla 🌿 Abrange as 3 aulas 📋 Requer template de PR
📋 Contexto e Objetivo

O projeto está desenvolvendo um dashboard para acompanhar o progresso das metas da ODS 11 (Cidades e Comunidades Sustentáveis). Sua tarefa é implementar o componente CardMeta — um card visual que exibe uma meta individual com seu indicador de progresso.

📁 Estrutura esperada:

projeto-ods/
├── index.html
├── styles.css
├── components/
│ └── CardMeta.html ← você cria
└── README.md

📌 Requisitos mínimos do card:

- Título da meta
- Descrição breve
- Indicador visual de progresso (% ou barra)
- Responsivo (funciona em mobile)
🚀 Passo a Passo
01
Clone o repositório
Clone o repositório compartilhado da turma (ou use o seu projeto-ods como base). Confirme que o git log --oneline mostra o histórico correto.
02
Crie a branch correta
Use o padrão: feature/SEU-NOME/card-meta-ods. Confirme com git branch que está na branch certa antes de começar a codar.
03
Implemente o CardMeta em 3 commits
Separe a implementação em exatamente 3 commits semânticos:
Commit 1: feat(card): cria estrutura HTML do CardMeta
Commit 2: style(card): adiciona estilos e barra de progresso
Commit 3: docs(card): documenta estrutura do componente
04
Publique a branch
Execute git push -u origin feature/SEU-NOME/card-meta-ods e confirme no GitHub que a branch aparece com os 3 commits.
05
Abra o PR com título profissional
Título: feat(dashboard): adiciona componente CardMeta com indicador de progresso. Adicione como base o branch main.
06
Preencha o template completo
Preencha todos os campos do template: descrição, checklist, issue relacionada, screenshot do card funcionando e notas para o revisor.
07
Adicione revisor e aguarde feedback
Adicione pelo menos 1 colega como revisor. Não faça o merge antes de receber e responder ao feedback.
08
Responda ao feedback e itere
Para cada comentário recebido: faça a mudança localmente → git commit -m "fix(card): aplica feedback do review" → git push → responda o comentário no GitHub explicando o que fez.
09
Aguarde aprovação e merge
Após a aprovação e o CI passar (se configurado), faça o merge. Delete a branch após o merge. Celebre! 🎉

📊 Critérios de Avaliação
🔧
Git — Técnico
  • Branch com nomenclatura correta
  • Exatamente 3 commits semânticos
  • Nenhum commit com mensagem genérica
  • Push na branch certa, não no main
🌐
GitHub — Processo
  • Título de PR no padrão Conventional
  • Template completamente preenchido
  • Screenshot incluída
  • Revisor adicionado corretamente
🤝
Colaboração
  • Respondeu a todos os comentários
  • Feedback incorporado com novos commits
  • Conversas do review marcadas como resolvidas
  • Merge apenas após aprovação
💡 Dicas para o Sucesso
🎯 Tamanho ideal de PR
Um bom PR tem entre 50–300 linhas alteradas. Se estiver muito grande, considere dividir em dois PRs menores.
📝 Mensagem de commit no review
Ao fazer ajustes do review, use: fix(componente): aplica feedback — descrição do que mudou
🔄 Mantenha a branch atualizada
Se demorar para o PR ser aprovado, execute git pull origin main na sua branch para pegar as atualizações antes do merge.
📸 Screenshots fazem diferença
Reviewers aprovam PRs visuais muito mais rápido quando há antes/depois ou print funcionando. Tire um print e cole diretamente no campo de texto do GitHub.
