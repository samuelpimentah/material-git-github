Conceito Importante

Git ≠ GitHub

🔧 Git
  • Ferramenta de controle de versão local
  • Roda na sua máquina, sem internet
  • Criado por Linus Torvalds (2005)
  • Software open source, gratuito
  • Gerencia commits, branches, merges
🌐 GitHub
  • Hospedagem de repositórios Git na nuvem
  • Plataforma de colaboração em equipe
  • Criado em 2008, adquirido pela Microsoft (2018)
  • Adiciona: Pull Requests, Issues, Actions, Projects
  • Alternativas: GitLab, Bitbucket, Azure DevOps
💡 Analogia

Git é o motor do carro. GitHub é o GPS, estacionamento e serviço de carona junto — ferramentas que tornam a viagem muito mais prática para equipes.

📌 Criando no GitHub primeiro?

Se o repositório já existe no GitHub, use git clone URL — ele já configura o remote automaticamente. O git remote add é para quando você já tem um repo local e quer publicá-lo.

Recebendo Atualizações

fetch vs pull — Qual a diferença?

git fetch — "mostra sem aplicar"

git fetch
$git fetch origin
From https://github.com/org/projeto-ods
a1b2c3d..e4f5g6h main → origin/main
$git log HEAD..origin/main --oneline
e4f5g6h feat(auth): login com Google
fetch + inspect

Baixa commits sem alterar sua branch local. Você inspeciona antes de aplicar — ideal quando quer ver o que mudou.

git pull — "busca e aplica"

git pull
$git pull
Updating a1b2c3d..e4f5g6h
Fast-forward
auth.js | 30 ++++++++++
pull = fetch + merge

Baixa E aplica imediatamente. Pode gerar conflitos se você tiver commits locais que não foram publicados ainda.

🎯 Recomendação

Comece o dia de trabalho com git pull na branch main. Antes de abrir um PR, faça git pull na sua feature branch para pegar as últimas atualizações.

Colaboração no GitHub

O que é um Pull Request?

Um PR é uma solicitação para mesclar sua branch no main. Mas mais do que isso — é um espaço de revisão e discussão do código.

  • Ninguém merge código no main diretamente — tudo passa por revisão
  • Colegas podem comentar linha a linha, sugerir mudanças
  • CI/CD (GitHub Actions) roda testes automáticos no PR
  • Exige aprovações mínimas antes do merge
  • Cria um registro permanente da decisão e justificativa
🎯 Por que "Pull Request"?

Você está pedindo ao dono do repositório: "Por favor, faça um pull das minhas mudanças para dentro do seu código." O nome vem dessa ideia original.

🌱 No Projeto ODS

Nenhum código entra no main sem ao menos 1 aprovação de um colega do 2º ano. O PR é a porta de entrada obrigatória.

Passo a Passo

Fluxo completo de um Pull Request

01
git switch -c feature/nova-funcionalidade
Cria a branch dedicada para a funcionalidade.
02
git commit -m "feat: implementa nova funcionalidade"
Desenvolve e cria commits semânticos na branch.
03
git push -u origin feature/nova-funcionalidade
Publica a branch no GitHub para que o PR possa ser criado.
04
GitHub → Compare & pull request → Preencher template
Cria o PR com título semântico, descrição seguindo o template da organização e revisores.
05
Code Review → Responder comentários → Push de ajustes
Revisores comentam; você faz ajustes com novos commits na mesma branch.
06
Aprovação → Squash & Merge → Branch deletada
Após aprovação e CI passando, o merge é feito e a branch é limpa.
Boas Práticas

Anatomia de um bom PR

  • Título semântico: segue Conventional Commits — feat(mapa): adiciona camadas de ODS
  • Tamanho ideal: PRs pequenos e focados são revisados mais rápido e com mais qualidade. Evite PRs "monstro".
  • Descrição completa: o quê mudou, por quê, como testar, screenshots se for visual.
  • Revisores: mínimo 1, idealmente alguém do 2º ano para o projeto ODS.
  • Labels: feature, bug, docs, WIP — para organizar e filtrar PRs abertos.
  • Linked Issue: referencia a task ou issue relacionada com Closes #42.
Título de PR — exemplos
✓ feat(mapa): adiciona camadas ODS interativas
✓ fix(auth): corrige token expirado no login
✓ docs: atualiza guia de contribuição
✗ atualiza coisas
✗ DONE
✗ fix bug
⚠️ WIP PRs

Use Draft Pull Request para sinalizar que o trabalho ainda está em progresso. Isso evita merge prematuro e ainda permite feedback antecipado.

Revisão de Código

Code Review — Colaborando na qualidade

Tipos de feedback no GitHub:

A
ana-2ano · Solicitação de mudança
Essa função ficou muito longa. Você consegue extrair o cálculo para um método separado? Facilita os testes e a leitura.
C
carlos-1ano · Comentário
Boa ideia usar async/await aqui em vez de callback. Muito mais legível!
P
pedro-2ano · Aprovado ✓
LGTM! Código limpo e os testes cobrem os casos principais. Aprovado para merge.

Como responder ao review:

  • Leia todos os comentários antes de sair mudando tudo
  • Faça as mudanças solicitadas e dê git push — os commits aparecem no PR
  • Responda cada comentário explicando o que fez ou por que discorda
  • Marque comentários resolvidos com "Resolve conversation"
  • Se discordar, argumente com educação — o revisor pode ter razão
🤝 Cultura de review

Review não é crítica à pessoa, é cuidado com o código. Seja construtivo ao revisar e aberto ao receber feedback.

Automação

GitHub Actions — CI no seu PR

GitHub Actions é a plataforma de automação do GitHub. Permite executar tarefas automáticas quando eventos acontecem no repositório — como a abertura de um PR.

  • Arquivo YAML em .github/workflows/
  • on: pull_request → roda ao abrir ou atualizar um PR
  • Garante que o código não quebra antes do merge
  • No projeto ODS: lint, testes e build automáticos
# .github/workflows/pr-check.yml
name: PR Check

on:
  pull_request:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install deps
        run: npm install
      - name: Run tests
        run: npm test
✅ Status no PR

O resultado do workflow aparece no PR com ✓ passing ou ✗ failing. Branches protegidas só permitem merge após CI passar.
