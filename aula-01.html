Contexto

O que é Git?

🎮

Pense como um jogo com checkpoints

Em um RPG, você salva o progresso antes de uma batalha difícil. Se morrer, volta ao save. O Git faz exatamente isso com o seu código: cada commit é um checkpoint que você pode visitar ou restaurar a qualquer momento.

  • Sistema de Controle de Versão (VCS): rastreia cada alteração feita em arquivos ao longo do tempo, com autor e data.
  • Distribuído: cada desenvolvedor tem uma cópia completa do histórico — sem dependência de servidor central.
  • Criado em 2005 por Linus Torvalds para o kernel Linux. Hoje é o padrão absoluto da indústria.
Contexto do Projeto

Por que isso importa no projeto ODS?

Vocês e os alunos do 2º ano vão colaborar em um projeto real vinculado às ODS da ONU. Isso significa múltiplos desenvolvedores no mesmo código.

  • Rastrear quem alterou o quê e quando
  • Trabalhar em funcionalidades paralelas sem conflito
  • Reverter erros sem perder o trabalho da equipe
  • Histórico claro de evolução do projeto

Configuração Inicial

Seu cartão de visita no Git

Antes de tudo, o Git precisa saber quem é você. Essa configuração fica salva globalmente e aparece em cada commit que você fizer.

bash — configuração global
$git config --global user.name "Seu Nome Completo"
$git config --global user.email "seu@email.com"
$git config --global core.editor "code --wait"
$git config --list
user.name=Seu Nome Completo
user.email=seu@email.com
core.editor=code --wait
💡 Por que --global?

Aplica a configuração para todos os projetos na sua máquina. Sem o flag, vale só para o repositório atual.

Conceito Fundamental

Os 3 estados de um arquivo

Todo arquivo no Git passa por três áreas antes de virar um checkpoint permanente.

📝
Working Directory
Onde você edita. Mudanças ainda não rastreadas pelo Git.
git add
📦
Staging Area
Sala de espera. Arquivos selecionados para o próximo commit.
git commit
🔒
Repository
Histórico permanente. Checkpoint salvo com hash único.
🎯 Por que ter Staging Area?

Ela permite escolher exatamente quais mudanças incluir em cada commit, mesmo tendo alterado vários arquivos. Você pode preparar uma mensagem de commit precisa e coesa.

Primeiros Comandos

Iniciar e inspecionar

Criando um repositório do zero:

git init
$git init
Initialized empty Git repository in .git/
📁 A pasta .git

O Git cria uma pasta oculta .git/ onde guarda todo o histórico. Nunca delete essa pasta!

Verificando o estado atual (use sempre):

git status
$git status
On branch main
Untracked files:
(use "git add" to track)
  index.html
  styles.css

💡 git status é o comando mais usado no dia a dia. Execute-o constantemente.

Staging Area

git add — Preparando o checkpoint

Move arquivos do Working Directory para a Staging Area. Pense como selecionar o que vai na mochila antes de fechar o zíper.

bash — variações do git add
$git add index.html
# Adiciona um arquivo específico
$git add src/
# Adiciona uma pasta inteira
$git add .
# Adiciona TUDO que mudou no diretório atual
$git status
Changes to be committed:
  new file: index.html
  new file: styles.css
⚠️ Cuidado com git add .

Adiciona tudo, incluindo arquivos com senhas e configs locais. Sempre use .gitignore para excluir o que não deve ser versionado (ex: .env, node_modules/).

Checkpoint

git commit — Salvando para sempre

O commit cria um snapshot permanente do staging, com autor, data e mensagem descritiva.

git commit
$git commit -m "feat: adiciona página inicial"
[main a1b2c3d] feat: adiciona página inicial
 2 files changed, 45 insertions(+)
$git log --oneline
a1b2c3d (HEAD → main) feat: adiciona página inicial

O que cada commit guarda:

  • Hash SHA-1: ID único e imutável (ex: a1b2c3d)
  • Autor e data: quem fez e exatamente quando
  • Mensagem: o que foi feito e por quê
  • Snapshot: estado completo dos arquivos
  • Pai: referência ao commit anterior, formando a cadeia
✅ Boa Prática

Commits pequenos e frequentes. Uma mudança lógica por commit — facilita encontrar bugs e entender a história.

Boas Práticas

Por que Commits Semânticos?

❌ Sem padrão — inútil para a equipe:

git log caótico
a1b2c3d ajustes
e4f5g6h corrigido
7h8i9j0 wip
k1l2m3n mudanças
n4o5p6q finalmente funcionou

✅ Conventional Commits — leitura instantânea:

git log semântico
a1b2c3d feat(auth): login com Google
e4f5g6h fix(form): valida email vazio
7h8i9j0 docs: atualiza README
k1l2m3n style: formata com Prettier
n4o5p6q chore: configura ESLint
🎯 Conventional Commits

Convenção da indústria que torna o histórico legível, permite gerar changelogs automáticos e facilita code review. Adotada por Vue.js, Angular, Electron e muitos outros projetos Open Source.

Desfazendo Alterações

git restore — Cancelar antes de commitar

Desfaz mudanças não commitadas. Perfeito para cancelar uma edição errada antes de criar o commit.

Descartar edição no Working Directory:

desfazer edição
$git restore index.html
# Arquivo volta ao estado do último commit

Remover arquivo da Staging Area:

remover do staging
$git restore --staged index.html
# Saiu do staging, mas a edição no arquivo permanece
⚠️ Operação irreversível

git restore sem --staged descarta as mudanças permanentemente. O Git não tem como recuperar — a edição simplesmente desaparece. Use com consciência.

Desfazendo Commits

git reset — Voltando no tempo

git reset HEAD~1 desfaz o último commit. A diferença está em o que acontece com as suas mudanças:

--soft
$git reset --soft HEAD~1
Desfaz o commit, mantém tudo na Staging Area. Pronto para criar um novo commit corrigido.
✓ Staging Area mantida
✓ Working Directory ok
--mixed (padrão)
$git reset HEAD~1
Desfaz o commit e limpa a Staging Area. Mudanças ficam no Working Directory para editar.
✗ Staging Area limpa
✓ Working Directory ok
--hard
$git reset --hard HEAD~1
Desfaz o commit e apaga tudo. Código desaparece. Use somente quando tem certeza absoluta.
✗ Staging Area limpa
✗ Working Directory apagado
