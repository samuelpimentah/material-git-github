delta-app-ofc

A organização Delta no GitHub

Uma GitHub Organization é como uma "empresa" no GitHub — agrupa múltiplos repositórios com membros, permissões e regras de workflow compartilhados.

delta-app-ofc (organização)
├── .github ← regras globais da org
│   ├── workflows/
│   ├── scripts/
│   ├── profile/
│   └── pull_request_template.md
├── delta-handbook ← documentação técnica
│   ├── DADOS/ DEVOPS/ EQS/
│   └── GERAL/ UX/
└── outros repos do projeto...
  • O repositório .github é especial: qualquer arquivo colocado nele (workflows, templates) vale para todos os repos da org automaticamente.
  • O pull_request_template.md na raiz do .github aparece pré-preenchido automaticamente sempre que alguém abre um PR.
  • Workflows definidos em .github/workflows/ podem ser reaproveitados entre repositórios.

O template de PR da organização

Todo PR aberto em qualquer repositório da Delta já vem com esse template preenchido automaticamente. Nenhum campo é opcional.

📋 pull_request_template.md · delta-app-ofc

## 📄 Descrição

> 2° ANO: registre o RF atendido.

Seções do template:

  • Descrição: o que foi implementado — 2º ano referencia o RF do doc de EQS
  • Tipo de PR: checkbox com os tipos Conventional Commits
  • Módulos 1º Ano: Backend Java / Frontend HTML / Database SQL
  • Módulos 2º Ano: Spring Boot, React/TS, Mobile, IA, BD, Infra, Docs/UX
  • Uso de IA: transparência obrigatória — se usou, documenta no Memorial
  • Padrões e Testes: Design Patterns e TDD utilizados
  • Checklist Técnico: 10 critérios de qualidade a marcar
⚠️ PR incompleto = bloqueado

Um Action da organização pode verificar automaticamente se o checklist está preenchido antes de permitir o merge.

Template · Transparência

Declaração de Uso de IA

O template exige que você declare se usou IA para gerar ou ajudar no código deste PR:

🤖 Uso de Inteligência Artificial
- [ ] Este código teve auxílio de IA.
- [ ] Este código foi feito 100% pelos integrantes.

> 1° ANO: Caso tenha usado IA,
> lembre-se de registrar o prompt e
> a validação crítica no Memorial Descritivo!
📝 O que é o Memorial Descritivo?

É um documento do 1º ano onde vocês registram o processo de desenvolvimento: decisões tomadas, ferramentas usadas, prompts de IA e como validaram criticamente o resultado.

Por que isso existe?

  • Honestidade intelectual: a equipe sabe o que foi gerado por IA e o que foi escrito pela turma
  • Responsabilidade técnica: IA gera código incorreto com frequência — você é responsável por validar tudo que commita
  • Aprendizado real: copiar sem entender não te prepara para o mercado de trabalho
  • Documentação do processo: o Memorial serve como portfólio de decisões do projeto
⚠️ IA não é proibida

IA é uma ferramenta, não um atalho. Use-a, mas entenda o código gerado, teste-o, adapte-o ao projeto e declare no template.

Commits na Org Delta

Conventional Commits: sua prioridade #1

O checklist técnico coloca Conventional Commits como primeiro item — é o requisito mais básico. PR com commit ruim não passa de review.

Commits válidos na Delta
✓feat: adiciona card de meta ODS
✓fix: corrige constraint NOT NULL na tabela metas
✓docs: atualiza RF-05 no handbook
✓refactor: extrai DAO para camada separada
✗atualizações
✗corrige bug
✗WIP
✗Adiciona Funcionalidade Nova

Regras específicas da Delta:

  • Descrição em minúsculas, sem ponto final, no imperativo
  • Mínimo 1 commit por lógica concluída — não commite código quebrado
  • Se um Action de CI detectar commits fora do padrão, o PR pode ser bloqueado automaticamente
🤖 Lint de commits automático

Alguns repos usam o commitlint via GitHub Actions para rejeitar automaticamente commits fora do padrão Conventional. Você vê isso no status ✓/✗ do PR.

🔑 Branch Protection Rules

O main da org tem branch protection ativa: não é possível fazer push direto — 100% do código entra pelo fluxo de PR + review + CI. Nem os admins quebram essa regra.

Exemplo Real · .github/workflows

Anatomy de um workflow da Delta

Estrutura típica de um workflow de CI para PRs:

# .github/workflows/pr-check.yml
name: PR Check · Delta

on:
  pull_request:
    branches: [ main ]

jobs:
  commit-lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - name: Lint commits
        uses: wagoid/commitlint-github-action@v5

  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install & Test
        run: npm ci && npm test

Lendo o workflow:

  • on: pull_request → o workflow roda toda vez que um PR é aberto ou atualizado mirando o main
  • fetch-depth: 0 → baixa o histórico completo, necessário para o commitlint checar todos os commits do PR
  • wagoid/commitlint-github-action → Action da comunidade que verifica Conventional Commits automaticamente
  • Dois jobs paralelos: lint de commits + build/testes
  • Ambos precisam passar (✓) para o merge ser liberado

Passo a Passo · Projeto Real

Como você vai contribuir no projeto Delta

01
Clone o repo da org e configure o remote:
git clone https://github.com/delta-app-ofc/REPO
02
Crie sua branch com o padrão da org:
git switch -c feature/descricao-curta
03
Desenvolva com commits semânticos frequentes e pequenos
04
Antes do PR: git pull origin main para evitar conflitos grandes
05
Push da branch e abra o PR:
git push -u origin feature/descricao
06
Preencha o template completamente — todos os campos
07
Aguarde CI e responda ao review dos membros

Nomenclatura de branches na Delta:

feature/card-meta-ods
↑ nova funcionalidade
fix/calculo-indicador
↑ correção de bug
docs/guia-contribuicao
↑ documentação
feature/Funcionalidade Nova
↑ inválido: maiúsculas e espaços
fix_bug
↑ inválido: sem prefixo, sublinhado
🚫
Nunca faça push direto no main. A branch protection bloqueia, mas a tentativa gera conflito e constrangimento. Sempre branch + PR.
delta-handbook · Documentação

Como contribuir com o Handbook

O handbook também segue o mesmo fluxo de PR. Ao adicionar ou atualizar documentação técnica:

exemplo de contribuição ao handbook
$git switch -c docs/modelo-bd-metas
# cria o arquivo na pasta correta
$touch DADOS/modelo-bd-metas.md
$git add DADOS/modelo-bd-metas.md
$git commit -m "docs: adiciona modelo BD das metas ODS"
$git push -u origin docs/modelo-bd-metas
# Abre PR → preenche template → review → merge

Antes de criar qualquer documento:

  • Verifique se já existe conteúdo semelhante no handbook — evite duplicatas
  • Escolha a pasta correta: DADOS, DEVOPS, EQS, GERAL ou UX
  • Use kebab-case minúsculo no nome do arquivo
  • Comece com título, objetivo e contexto claros
  • Use diagramas (Mermaid, imagens) quando o texto não for suficiente
  • Documente decisões e justificativas, não só o resultado
📐 Dica: Referências ao RF

Se seu documento está ligado a um Requisito Funcional (RF) do projeto, referencie-o claramente: RF-05: Exibição de Indicadores. Isso conecta código, docs e testes.
